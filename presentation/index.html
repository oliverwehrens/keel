A pipeline of gated, human-reviewed artifacts — not a magic "build my app" button.

The status quo

How we build today

  • Humans type the code; agents autocomplete
  • The spec lives in tickets, threads, and heads
  • Throughput bounded by typing & reading speed
  • Tests written after, if at all

The implicit bet

  • The expensive, scarce resource is writing code
  • So tools optimize keystrokes and navigation

That bet is what changes.

What agentic engineering changes

  • Agents now plan, write, test, and review code — not just complete a line
  • Typing stops being the bottleneck; clarity of intent becomes it
  • The specification — not the keystroke — is the scarce input
The constraint moves from "how fast can we write code?" to "how precisely can we say what we want — and verify we got it?"

How it changes development

The developer becomes…

  • A director of intent
  • An architect of constraints
  • A reviewer & verifier

The work becomes…

  • A pipeline of gated artifacts
  • Acceptance criteria as first-class drivers
  • Verification is the deliverable, not an afterthought

More leverage — but output quality is capped by spec quality and verification.

Coding → Specification

An agent is rarely blocked by typing. It is blocked by ambiguity.

  • Vague intent → confidently wrong code
  • Precise intent → faithful, reviewable output
  • So effort moves upstream: define, refine, agree — then generate
The specification is the new source code. Code becomes a build artifact of a well-formed spec.

Why split product & implementation

  • The source of truth must be the product (what & why) — never the code
  • Reaching confidence on a paragraph is far cheaper than on a codebase
  • Agents only implement faithfully from a solid, unambiguous spec
  • One product spec can drive any tech — and tech traces back to it
  • Two modes of thinking: divergent product vs convergent execution
  • A hard gate (reconcile) finalizes product before a single tech decision
Decide what until you're confident. Only then decide how.

The loop

Product briefing→ epic→ refine→ reconcile ★ source of truth
Tech tech-stack→ tech-refine · traces to product
Build breakdown→ implement→ verify
  • Artifact-driven: state lives in markdown, not chat — resumable & reviewable
  • Gated: nothing advances until it's earned it
  • Human-in-the-loop: open questions posed as 3 options, you pick
Product

sdd-briefing

Capture the raw intent — the seed every epic grows from.

∅ → specs/briefing.md
  • Problem · users · goals · non-goals · constraints · success metrics
  • Interviews you; presses on non-goals and outcome-based metrics
Honest and short beats long and vague. Unknowns are parked, not faked.
Product

sdd-epic

Turn the briefing into coherent, right-sized slices of value.

briefing.md → specs/epics/EPIC-NNN.md
  • Each epic: scope (in/out), capabilities, Given/When/Then criteria
  • Good first drafts — not finished epics
Acceptance criteria are written as scenarios — the seed of an executable test.
Product

sdd-refine

Iterate one epic until it clears the confidence gate.

draft epic → ready epic
  • Every open question → 3 options, pros/cons, one (recommend) — you pick [x]
  • Won't mark ready until scope, criteria, questions & risks all pass
Confidence is earned and logged — never faked to pass the gate.
Product

sdd-reconcile

Make a ready epic final — and consistent.

ready epic → reconciled (final)
  • Vertical: traces to briefing, violates no non-goal, covers the goals
  • Horizontal: no overlap / contradictions across epics; consistent terms
Output is the source of truth all tech & code must conform to.
Tech

sdd-techstack

Choose the stack — only after the product is settled.

reconciled epics → specs/tech-stack.md
  • Every choice traces to a product need; alternatives recorded
  • Genuine forks posed as 3-option decisions
No choice without a driver. Tech may not invent product requirements.
Tech

sdd-tech-refine

Design the architecture on that stack; refine to a tech gate.

tech-stack.md → specs/architecture.md
  • Principles, component boundaries, cross-cutting concerns
  • Key decisions as immutable, numbered ADRs; guardian-reviewed
Decides how, never what. The product spec still wins.
Build

sdd-breakdown

Split a final epic into implementation stories.

reconciled epic → specs/stories/…
  • Vertical, independently verifiable slices; sequenced; DoD per story
  • Open questions routed by kind — product ones bounce back upstream
Breakdown decides slicing — it may never silently redefine the product.
Build

sdd-implement

Build each story test-first — London-school (mockist) TDD.

story → code + tests
  • Outside-in double loop · need-driven interface discovery
  • Mock roles you own; wrap third-party libs behind ports
  • Guardian gate on every task; does not self-certify
The failing acceptance test is the epic's Given/When/Then.
Build

sdd-verify

Independently confirm the work — separate from who built it.

in-progress story → verified
  • Runs the suite · criterion ↔ test ↔ code traceability · DoD
  • Epic fidelity: stories must add up to the epic — not just pass alone
Never marks verified on assertion — only on evidence.
Subagent · cross-cutting

architecture-guardian

A read-only reviewer that runs through tech-refine, implement & verify.

  • Checks: product fidelity → stack → architecture conformance
  • Reports blocker / warning / note → PASS or CHANGES REQUIRED
When spec and code conflict, the product spec wins — every time.
Beyond greenfield

Brownfield

Existing code, thin specs? Invert the flow.

code→ sdd-adopt · inferred spec→ ratify→ the loop
  • Just-in-time: baseline only the slice you're about to touch
  • inferred → ratified: reverse-engineered guesses aren't truth until a human signs off
  • Legacy-aware build: characterization tests · seams · sprout/wrap · strangler fig
Same gates, same source-of-truth discipline — established descriptively first.
Brownfield

sdd-adopt

Reverse-engineer the spec from existing code.

code → inferred spec → ratify
  • Just-in-time: baseline only the slice you'll touch
  • Emits inferred briefing, as-is epics, tech-stack, architecture (retro ADRs), glossary
  • Ratification gate: inferred → ratified before it's trusted
Code → spec → code. Descriptive first; nothing is truth until a human signs off.
Beyond greenfield

Iteration — re-enter, don't restart

After the first loop, a change is a delta — enter at the right phase.

  • New feature → epic · change → refine · refactor → tech-refine · bug → implement
  • sdd-impact predicts the blast radius — product epics first — before you change
  • Reconcile checks the delta against existing epics & the briefing
  • Non-regression at verify: every prior verified epic stays verified
  • sdd-audit catches spec ↔ code drift before you build
The loop is a cycle, not a waterfall. The spec stays the living source of truth.
Lifecycle

sdd-impact

Predict the blast radius of a change before you make it.

proposed change → impact report
  • Traverses across (sibling epics) · up (code → product epics) · down (stories/tests)
  • Leads with product epics affected — contradictions flagged first
  • Feeds non-regression & sequencing · read-only
A predictive reconcile — know the ripple before you touch anything.
Lifecycle

sdd-audit

Detect spec ↔ code drift on a living project.

codebase slice → drift report
  • Undocumented code · unbuilt spec · arch violations · stale traces · glossary drift
  • Reports & routes to the owning phase — never fixes silently
  • Verdict: IN SYNC / DRIFT FOUND
Keeps the source of truth actually true over time.

The loop, mechanically

run skill→it writes 3-option questions→you mark [x]→re-run / "continue"↺
  • Idempotent & resumable: rebuilt from artifacts, not chat memory
  • Held together by anchors: source of truth · Definition of Ready/Done · Given/When/Then · ADRs · ubiquitous language
Lose the session, switch machines — the markdown is the state. Nothing is lost.

What to watch out for

Garbage in

A vague spec yields confidently wrong code. Refine before you generate.

Over-trust

Unverified output isn't done. Verification is the deliverable.

Lost source of truth

Don't let code redefine intent. The product spec outranks everything.

Architectural drift

Many agent edits erode boundaries — hence a standing guardian.

Rubber-stamping

Review fatigue is real. Gates & traceability force evidence.

Scope creep

Agents do more than asked. "No scope drift" is an explicit check.

Takeaways

  • Agentic engineering moves the bottleneck from typing to specifying & verifying
  • Make the product the source of truth; let tech & code trace back to it
  • Reach confidence on what before committing to how
  • Keep state in artifacts, gate every step, and verify against the spec
The craft isn't gone — it moves up a level: from code to the specification that generates it.
